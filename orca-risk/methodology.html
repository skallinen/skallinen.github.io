Technical Methodology

Step-by-step documentation of the Bayesian analysis pipeline: data collection, encoding, model specification, diagnostics, and the time-of-day problem.

Companion to What Makes Orcas Interested in Your Boat? · June 2026

1. Data Collection

The Cruising Association maintains a public reporting system where skippers can file two types of reports: incident reports (an orca interaction occurred) and uneventful passage reports (the boat transited the affected zone without incident). Both forms collect the same set of variables.

We collected all reports via the Cruising Association's public JSON API. The API exposes a report list endpoint and individual report detail endpoints. An API client iterates through the report list, fetches each report's details concurrently, and stores them as structured JSON.

# API endpoints (public, no authentication required)
REPORT_LIST = "https://orcas.cruising.org.uk/api/v1/reports"
REPORT_DETAIL = "https://orcas.cruising.org.uk/api/v1/reports/{id}"

# Rate limiting: 0.5s delay between requests
# Output: all_reports_detailed.json (654 reports)

The dataset contains 654 reports: 216 incidents and 438 uneventful passages. Each report records 20+ fields covering vessel characteristics, environmental conditions, and location.

2. Data Preparation

Raw reports required substantial cleaning. The two report types use different field names for some variables (e.g. towing vs trailing, date_of_interaction vs date_passage_commenced). The preparation step harmonises these into a single modelling-ready dataset.

Variable encoding

Variables fall into three types, each encoded differently following McElreath's Statistical Rethinking (Ch. 5):

TypeVariablesEncoding
OrdinalDepth, distance, speed, boat length, wind, sea stateInteger scale (0, 1, 2, 3), then z-score standardised
CategoricalSailing mode, antifoul colour, hull colour, rudder typeIndex coded (0 to K-1), each category gets its own parameter
BinaryAutopilot on/off0/1

Ordinal standardisation

Ordinal predictors are mapped to integers (e.g. depth: <20m=0, 20-40m=1, 40-200m=2, 200m+=3), then standardised to zero mean and unit standard deviation. This puts all ordinal predictors on a comparable scale, which makes prior specification simpler: a Normal(0, 0.5) prior on a standardised predictor implies a belief that moving one standard deviation in the predictor changes the log-odds by at most about 1.

# Standardisation parameters (computed from full dataset)
depth:    mean=1.708, sd=0.951
distance: mean=1.355, sd=1.147
speed:    mean=1.939, sd=0.450
length:   mean=1.683, sd=0.815
wind:     mean=0.994, sd=0.804
sea:      mean=0.511, sd=0.601

Categorical index coding

For categorical variables with K levels, we estimate K separate intercept offsets (not K-1 dummy variables). This follows McElreath's recommendation: it avoids privileging one category as the reference, and the priors are symmetric across all levels.

Missing data

We use complete-case analysis. Any observation missing a value for any predictor in the model is excluded. This reduces the dataset from 654 to 605 observations. The missing rate is low for most variables (under 5%); the highest is rudder type at 7%.

3. Causal Structure

Before specifying the model, we encoded domain knowledge as a causal directed acyclic graph (DAG). Variables were organized into layers:

  • Layer 0 (Unobserved): Orca presence/activity (latent), location/route (partially observed via depth and distance)
  • Layer 1 (Vessel, pre-treatment): Boat type, boat length, rudder, antifoul colour, hull colour
  • Layer 2 (Activity): Sailing mode, speed, autopilot
  • Layer 3 (Environment): Wind, sea state, depth, distance off land
  • Layer 4 (Outcome): Interaction (binary)

The DAG identifies two critical issues. First, geographic location is a strong confounder (it affects both orca presence and boat characteristics) but is only partially observable through depth and distance. Second, post-outcome variables like damaged and tow_required are colliders and must never be conditioned on.

For predictive purposes (our primary goal), we include all observables. For causal interpretation, the DAG warns us that confounding by unobserved location means we can identify associations but not causal effects.

4. Model Specification

Likelihood

The outcome is binary: interaction (y=1) or uneventful (y=0). We use a Bernoulli likelihood with a logit link:

y_i ~ Bernoulli(p_i)
logit(p_i) = α + β_depth·depth_i + β_autopilot·autopilot_i + β_speed·speed_i
             + β_length·length_i + β_distance·distance_i + β_wind·wind_i
             + β_sea·sea_i + α_sailing[sailing_i] + α_antifoul[antifoul_i]
             + α_hull[hull_i] + α_rudder[rudder_i]

This is a standard logistic regression with 30 parameters: one intercept (α), seven continuous/ordinal coefficients (β), and 22 category-level offsets (5 sailing modes + 8 antifoul colours + 3 hull colours + 6 rudder types).

Priors

We follow the weakly informative prior strategy from Statistical Rethinking (Ch. 11). During model building (M0-M4), we used a Fermi-informed intercept prior. For the final refitted model (without daylight), we relaxed to a wider prior:

# Model building phase (M0-M4)
α ~ Normal(-3.5, 0.6)     # Fermi: logit(0.025) ≈ -3.66
                           # Spans ~1% to ~7% at ±2 SD

# Final refitted model (M3 without daylight)
α ~ Normal(-1, 1)         # Wider, less informative
                           # Lets data determine the intercept

# Ordinal/binary coefficients
β_depth     ~ Normal(0, 1)    # wider: depth expected to matter
β_autopilot ~ Normal(0, 1)    # wider: autopilot expected to matter
β_speed     ~ Normal(0, 0.5)  # standard regularising prior
β_length    ~ Normal(0, 0.5)
β_distance  ~ Normal(0, 0.5)
β_wind      ~ Normal(0, 0.5)
β_sea       ~ Normal(0, 0.5)

# Category offsets
α_sailing[k]  ~ Normal(0, 0.5)  for k = 1..5
α_antifoul[k] ~ Normal(0, 0.5)  for k = 1..8
α_hull[k]     ~ Normal(0, 0.5)  for k = 1..3
α_rudder[k]   ~ Normal(0, 0.5)  for k = 1..6

The sensitivity analysis confirmed that the slope coefficients (all the relative risk factors) are robust across different intercept priors. Only the absolute risk level changes. We tested four intercept priors ranging from Normal(-4.5, 0.6) to Normal(0, 1.5) and found negligible differences in slope posteriors.

Prior predictive simulation

Before fitting, we simulated predictions from the prior to verify that the prior puts probability mass on reasonable interaction rates. With these priors, the prior predictive distribution covers rates from near 0% to near 100%, centred around 25-35%, consistent with our pre-data expectations.

Model building sequence

We built five models of increasing complexity, following McElreath's recommendation to add predictors in stages:

ModelPredictorsParametersPurpose
M0Intercept only1Base rate
M1+ boat length, hull colour, rudder, antifoul19Vessel characteristics
M2+ sailing mode, autopilot, speed26Activity state
M3+ depth, distance, wind, sea state30Environmental conditions (primary model)
M4+ moon, tide, cloud cover33Extended model

M3 is the primary model used for all results. Models are compared by their expected log pointwise predictive density estimated with the WAIC (Widely Applicable Information Criterion), computed from the pointwise log-likelihood each model emits. M4 adds moon, tide, and cloud cover, which showed no credible effects and a slightly worse (indistinguishable, within one standard error) WAIC than M3. Time of day was originally in M3 but was removed after the encoding analysis described below.

5. MCMC Sampling

All models are written in Stan and fitted with CmdStan using the NUTS (No-U-Turn Sampler) algorithm, driven from Clojure:

# Sampling configuration
Chains:      4
Tune:        1,000 iterations per chain
Draw:        2,000 iterations per chain
Total draws: 8,000 (4 × 2,000)
Seed:        42 (for reproducibility)

All four chains converged cleanly for M3. Convergence is checked with diagnostics computed directly in Clojure following the standard Stan/ArviZ definitions: the rank-normalised, folded split-R-hat (Vehtari et al. 2021) and bulk/tail effective sample size (ESS). For M3 there were no divergences, R-hat values were all within 0.01 of 1.0, and ESS comfortably exceeded 400 for every parameter.

6. Case-Control Correction

The dataset contains 216 incidents and 438 uneventful passages, giving a sample interaction rate of 33%. The true population rate is much lower. We estimated the base rate at roughly 2-5% using a Fermi calculation:

  • ~15,000 yacht transits per year through the affected zone (AIS estimates)
  • ~300 reported interactions per year (GTOA data, 2020-2024 average)
  • Implied rate: ~2% (with substantial uncertainty)

Because incident reports are over-represented relative to the population, the model's intercept is biased upward. We correct this using the King & Zeng (2001) rare-events logistic regression offset:

offset = log(π / (1 - π)) - log(ȳ / (1 - ȳ))

where:
  π = true population base rate (user-adjustable, default 2.5%)
  ȳ = sample rate (0.330)

This offset is added to the logit prediction at inference time. Crucially, it only shifts the intercept. All slope coefficients (the relative risk factors) are unaffected by the case-control ratio. This is a well-known property of logistic regression: the odds ratios are consistent even when the sampling proportions do not match the population proportions.

7. Time-of-Day Analysis

Time of day was originally included as a binary predictor (day=1, night=0) in the logistic regression. It appeared to be the strongest predictor, with a night coefficient of +2.06 on the log-odds scale (OR 7.8×). We removed it after discovering that the finding was an artifact of how the two groups record time differently.

The encoding problem

Incident reports record a single moment: the skipper selects "Day," "Night," "Dawn," or "Dusk" to describe when the interaction occurred. Uneventful passage reports record a time span: start and end times that may cover multiple periods ("Dawn, Day, Dusk, Night" for a 24-hour passage).

Our initial encoding collapsed multi-period passages to "Day" (since they contained "Day" as one component). This left only 11 uneventful passages classified as night, creating an extreme class imbalance that inflated the night coefficient.

Sensitivity analysis

We tested three encoding strategies and found that the direction of the night effect depended entirely on the encoding choice:

EncodingNight rateDay rateModel result
A: Multi-period → day85.5%26.1%Night 7.8× riskier
B: Any "Night" → night21.1%36.3%Night 6× safer
C: Split into exposure windows27.1%28.1%No difference

Exposure-based Bayesian rate model

Rather than forcing the ambiguous time variable into the logistic regression, we analysed time of day separately using an exposure-based approach. Both datasets contain actual timestamps: incident reports have a time of interaction; uneventful passages have start and end times.

Using the commons-suncalc Java library, we computed the solar position for each timestamp at a central location in the orca zone (37°N, 8°W) and classified each moment as Day, Night, Dawn, or Dusk based on astronomical twilight boundaries.

For uneventful passages, we sampled the solar period at regular intervals between start and end times to compute total yacht-hours in daylight versus darkness. For incidents, we classified the single interaction timestamp.

# Bayesian Poisson rate model
λ_night ~ Exponential(50)     # prior mean ≈ 0.02 per yacht-hour
λ_day   ~ Exponential(50)

y_night ~ Poisson(λ_night × T_night)  # 50 incidents, 4,280 yacht-hours
y_day   ~ Poisson(λ_day × T_day)      # 166 incidents, 7,933 yacht-hours

# Derived quantities
rate_ratio = λ_night / λ_day

The posterior rate ratio is 0.56 (89% CI: 0.43 to 0.72), with 100% posterior probability that night has a lower incident rate. The result is insensitive to the prior (a flat Gamma(1,1) prior gives identical results).

Interaction tests

We extended the Poisson model to test whether risk factors behave differently at night versus day. The 4-rate model (e.g. night-black, night-other, day-black, day-other) showed no significant interactions:

  • Black antifoul effect: 2.1× day, 2.4× night (interaction ratio 1.2×, 89% CI spans 1.0)
  • Motoring effect: 4.1× day, 5.5× night (interaction ratio 1.4×, 89% CI spans 1.0)

Because the effects are stable across time of day, the calculator applies the overall day/night rate ratio as a uniform scaling factor.

8. Confound Analysis

For each strong predictor, we tested whether its effect could be explained by correlation with other covariates.

Autopilot

Bayesian comparison of group means showed that autopilot-on boats are slightly shallower (96% posterior probability) but the difference in distance from land is uncertain (92%, credible interval spans zero). The controlled odds ratio (3.0×) is larger than the raw odds ratio (2.3×), indicating that the covariates were suppressing the autopilot signal, not creating it.

Black antifoul

Black-antifoul boats sail in deeper water (+0.31 SD, 89% CI: +0.17 to +0.44) and farther offshore (+0.30 SD, 89% CI: +0.17 to +0.43), both with 100% posterior certainty. The raw odds ratio of 3.9 drops to a controlled posterior odds ratio of 1.5 after accounting for all other variables. The confounds explain part of the raw effect, but a credible association remains.

As an additional check, a single-predictor model with only antifoul colour yielded a Black vs Coppercoat contrast of +1.16 on the log-odds scale (89% CI: +0.72 to +1.60). The full model gives +1.00 (89% CI: +0.52 to +1.51), confirming the direction and strength are consistent with or without controls.

9. Posterior Export and Calculator

For the interactive calculator, we export 500 randomly sampled posterior draws (from the 8,000 total) as a JSON file. Each draw is a vector of 30 parameter values, preserving the full correlation structure between parameters.

# posterior_draws.json structure
{
  "n": 500,
  "layout": ["alpha", "b_depth", "b_autopilot", "b_speed",
             "b_length", "b_distance", "b_wind", "b_sea",
             "s_0"..."s_4", "a_0"..."a_7", "h_0"..."h_2", "r_0"..."r_5"],
  "categories": { ... },
  "standardization": { ... },
  "draws": [[0.123, 0.456, ...], ...]  // 500 × 30
}

The calculator runs each of the 500 draws through the logistic function with the user's selected inputs, applies the King & Zeng base-rate correction, applies the day/night rate ratio adjustment, and reports the median and 89% credible interval of the resulting 500 predictions.

The day/night adjustment is multiplicative. The logistic regression predicts for the average time-of-day mix in the training data (65% day, 35% night by yacht-hours). The adjustment factors are:

# Day/night multiplicative adjustment
# Rate ratio (night/day) = 0.56
# Exposure fractions: 65% day, 35% night
day_factor   = 1 / (0.65 + 0.35 × 0.56) = 1.182
night_factor = 0.56 × day_factor          = 0.662
